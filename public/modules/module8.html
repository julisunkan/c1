Module 8: Web Application Testing

Web application testing focuses on identifying security vulnerabilities in web-based applications and APIs. This module covers the OWASP Top 10, manual testing techniques, and automated scanning approaches for comprehensive web application security assessment.

Learning Objectives

  • • Master web application security testing methodologies and frameworks
  • • Understand and exploit OWASP Top 10 vulnerabilities
  • • Learn manual testing techniques for complex business logic flaws
  • • Practice API security testing and authentication bypass techniques
  • • Implement automated scanning and integration approaches
  • • Develop comprehensive web application testing strategies

OWASP Top 10 Web Application Security Risks

The OWASP Top 10 represents the most critical web application security risks that organizations face.

OWASP Top 10 - 2021

A01:2021 - Broken Access Control

Failures related to access control policy enforcement

A02:2021 - Cryptographic Failures

Previously known as Sensitive Data Exposure

A03:2021 - Injection

SQL, NoSQL, OS, and LDAP injection vulnerabilities

A04:2021 - Insecure Design

Design flaws and missing security controls

A05:2021 - Security Misconfiguration

Insecure default configurations and missing security hardening

SQL Injection Testing

SQL injection remains one of the most critical web application vulnerabilities, allowing attackers to manipulate database queries.

Manual Detection

' OR '1'='1 Basic authentication bypass
' UNION SELECT NULL-- Union-based injection
' AND 1=1-- Boolean-based blind
' AND (SELECT SLEEP(5))-- Time-based blind

Automated Tools

  • • SQLMap: Automated SQL injection testing
  • • NoSQLMap: NoSQL injection testing
  • • Commix: Command injection testing
  • • jSQL Injection: Java-based SQL injection tool
  • • Havij: Automated SQL injection tool
  • • BSQL Hacker: Blind SQL injection framework

Cross-Site Scripting (XSS)

XSS vulnerabilities allow attackers to inject malicious scripts into web applications viewed by other users.

XSS Types and Payloads

Reflected XSS

Script reflected back in response

<script>alert('XSS')</script>
Stored XSS

Script stored in application database

<img src=x onerror="alert('Stored XSS')">
DOM XSS

Client-side DOM manipulation

javascript:alert('DOM XSS')

Authentication and Session Management

Weak authentication and session management can lead to account compromise and unauthorized access.

Authentication Flaws

  • • Weak Password Policy: Simple password requirements
  • • Brute Force Attacks: No account lockout mechanisms
  • • Credential Stuffing: Reused password attacks
  • • Password Reset Flaws: Insecure reset mechanisms
  • • Multi-Factor Bypass: MFA implementation flaws
  • • Session Fixation: Predictable session identifiers

Session Management Issues

  • • Weak Session IDs: Predictable or short tokens
  • • Session Hijacking: Token interception attacks
  • • Insecure Transmission: Unencrypted session data
  • • Session Timeout: Inadequate timeout policies
  • • Concurrent Sessions: Multiple active sessions
  • • Logout Failures: Incomplete session termination

Business Logic Testing

Business logic flaws arise from design and implementation errors in application workflows and processes.

Common Business Logic Flaws

Price Manipulation

Modifying product prices during checkout process

Workflow Bypass

Skipping mandatory steps in business processes

Race Conditions

Concurrent request processing vulnerabilities

Privilege Escalation

Gaining higher privileges through application flaws

API Security Testing

Modern applications heavily rely on APIs, making API security testing a critical component of web application assessments.

REST API Testing

  • • HTTP Methods: Test all supported methods
  • • Parameter Pollution: Duplicate parameter handling
  • • Input Validation: Boundary and format testing
  • • Rate Limiting: DoS and brute force protection
  • • CORS Issues: Cross-origin resource sharing
  • • Error Handling: Information disclosure in errors

GraphQL Testing

  • • Introspection: Schema information disclosure
  • • Query Complexity: Resource exhaustion attacks
  • • Authorization: Field-level access controls
  • • Injection Attacks: GraphQL-specific injections
  • • Batching Attacks: Query batching vulnerabilities
  • • Alias Attacks: Query alias abuse

File Upload Vulnerabilities

File upload functionality presents numerous security risks if not properly implemented and validated.

File Upload Attack Vectors

Malicious File Upload

Upload web shells and executable files

MIME Type Spoofing

Bypass content-type restrictions

Directory Traversal

Upload files to arbitrary locations

Size and Resource Abuse

Upload large files to exhaust storage

Client-Side Security Testing

Client-side security focuses on vulnerabilities in browser-based code and local storage mechanisms.

JavaScript Security

  • • Code Obfuscation: Reverse engineering protection
  • • Sensitive Data Exposure: Hardcoded secrets
  • • Client-Side Validation: Bypass attempts
  • • Prototype Pollution: JavaScript object manipulation
  • • WebAssembly: Binary code analysis
  • • Source Map Exposure: Development artifact leakage

Browser Storage

  • • Local Storage: Persistent data exposure
  • • Session Storage: Temporary data leakage
  • • Cookie Security: HttpOnly and Secure flags
  • • Web SQL: Client-side database injection
  • • IndexedDB: Structured storage vulnerabilities
  • • Cache Poisoning: Browser cache manipulation

Manual Testing Techniques

Manual testing identifies complex vulnerabilities that automated tools often miss, especially business logic flaws.

Proxy-Based Testing

  • • Request Interception: Modify requests in transit
  • • Response Analysis: Examine server responses
  • • Session Tracking: Monitor authentication state
  • • Parameter Fuzzing: Input validation testing
  • • History Analysis: Review request patterns
  • • Cookie Manipulation: Session token modification

Browser-Based Testing

  • • Developer Tools: Network and storage inspection
  • • Console Injection: JavaScript execution
  • • Source Code Review: Client-side code analysis
  • • Local Storage: Data persistence examination
  • • Form Manipulation: Hidden field modification
  • • CSRF Testing: Cross-site request forgery

Web Application Firewalls (WAF) Bypass

Understanding WAF bypass techniques helps test the effectiveness of web application protection mechanisms.

Bypass Techniques

Encoding and Obfuscation

URL encoding, Unicode normalization, case variation

HTTP Parameter Pollution

Duplicate parameters with different interpretations

Request Smuggling

HTTP request parsing differences

Whitelist Exploitation

Abuse allowed patterns and functions

Reporting and Documentation

Comprehensive reporting ensures vulnerabilities are properly communicated and remediated effectively.

Vulnerability Documentation

  • • Description: Clear vulnerability explanation
  • • Impact: Business and technical consequences
  • • Steps to Reproduce: Detailed exploitation steps
  • • Proof of Concept: Working exploit demonstration
  • • Risk Rating: CVSS or custom scoring
  • • Remediation: Specific fix recommendations

Executive Summary

  • • Risk Overview: High-level security posture
  • • Critical Findings: Most severe vulnerabilities
  • • Business Impact: Operational risk assessment
  • • Recommendations: Priority remediation actions
  • • Compliance Status: Regulatory requirement gaps
  • • Timeline: Suggested remediation schedule

Key Takeaways

Web application testing requires a combination of automated scanning and manual testing techniques to identify the full spectrum of security vulnerabilities. This module covered the OWASP Top 10, manual testing methodologies, API security assessment, and business logic testing. Effective web application testing demands understanding both technical vulnerabilities and business context to provide meaningful security assessments. Remember that automated tools are starting points for deeper manual investigation, and the most critical vulnerabilities often require human insight to discover and exploit properly.
